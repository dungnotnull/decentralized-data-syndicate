# PROJECT-detail.md — decentralized-data-syndicate

## Executive Summary

The **Decentralized Data Syndicate** is a peer-to-peer marketplace where autonomous AI agents negotiate, verify, and transact data assets using smart contracts and Zero-Knowledge Proofs. Buyers broadcast structured data-need signals across a Libp2p network; matching seller agents surface relevant datasets, present human-approved offers, and transfer end-to-end encrypted data only after escrow is confirmed. A ZK-SNARK circuit validates statistical quality properties (schema compliance, completeness, entropy) without exposing raw content — ensuring buyers get what they paid for without compromising seller privacy. The entire protocol layer is deterministic and cryptographic; natural language LLMs are used only to translate buyer queries into structured protocol messages.

---

## Problem Statement

### The Broken Data Economy

The global data broker market was valued at **$268 billion in 2023** and is projected to reach **$545 billion by 2031** (Grand View Research). This value is almost entirely captured by centralized intermediaries — data brokers, cloud providers, and platform monopolies — while the actual data producers (individuals and their devices) receive nothing. Key pain points:

- **For data sellers (individuals)**: No mechanism to monetize personal behavioral data; no transparency about who uses it or why; no ability to audit quality claims before sale.
- **For data buyers (AI trainers, researchers)**: High-quality domain-specific datasets are expensive, often synthetic, and frequently of unknown provenance. Academic and indie researchers are priced out.
- **For both parties**: Centralized platforms are single points of failure, censorship, and data exfiltration.
- **For privacy**: Every existing data marketplace requires the seller to expose raw data to a central validator before payment — destroying the privacy guarantee.

### Why ZK-Proofs Change Everything

Zero-Knowledge Proofs allow a seller to *prove* statistical facts about a dataset (e.g., "this CSV has 10,000 rows, <2% null values, and schema matching your spec") without revealing any actual data values. This decouples quality assurance from privacy exposure — the core technical innovation of this project.

## ML/DL Models

### 1. Local Data Quality Scorer
- **Model**: `microsoft/phi-3-mini-4k-instruct` via Ollama
- **Task**: Given a data sample (first 100 rows), produce a structured quality report: schema inference, null rate per column, duplicate row percentage, semantic coherence score
- **Input**: CSV/JSONL snippet (never more than 200 rows for privacy)
- **Output**: `{ "schema_match": true/false, "null_rate": 0.012, "duplicate_rate": 0.003, "coherence_score": 0.87, "recommendation": "proceed|reject" }`
- **Fine-tuning**: Not required initially; prompt engineering with few-shot examples is sufficient
- **Training data**: If fine-tuning needed: synthetic datasets with known quality metrics from Kaggle and UCI ML Repository

### 2. Semantic Data-Need Matcher
- **Model**: `sentence-transformers/all-MiniLM-L6-v2`
- **Task**: Convert buyer's DataNeedSpec description and seller's data catalog entry into dense vectors; cosine similarity for relevance ranking
- **Why**: Structured keyword matching misses semantic equivalence (e.g., "NestJS usage logs" vs "Node.js backend API call traces")
- **Fine-tuning**: Not required; off-the-shelf model is sufficient for general-purpose embedding

### 3. ZK Circuit (Not ML — Deterministic)
- **Framework**: circom 2.0
- **Proven Facts** (public outputs of the proof):
  - Row count ≥ declared minimum
  - Null rate per column ≤ declared maximum
  - Schema column names/types match declared spec (hash comparison)
  - Entropy score ≥ threshold (prevents all-identical-value padding attacks)
- **Proving System**: Groth16 (fast verification, suitable for on-chain verifier)
- **On-chain verifier**: Auto-generated Solidity verifier from snarkjs

## External LLM API Integration

The protocol layer is fully deterministic. LLMs serve only one role: translating a buyer's natural language data need into a structured `DataNeedSpec` Protobuf message.

```python
# Pluggable LLM backend — config: LLM_PROVIDER=claude|openai|ollama
class QueryTranslator:
    def translate(self, nl_query: str) -> DataNeedSpec:
        prompt = TRANSLATE_PROMPT_TEMPLATE.format(query=nl_query)
        match LLM_PROVIDER:
            case "claude":
                return call_claude_api(prompt)  # CLAUDE_API_KEY
            case "openai":
                return call_openai_api(prompt)  # OPENAI_API_KEY
            case "ollama":
                return call_ollama(prompt)       # OLLAMA_BASE_URL (default)
```

If LLM translation fails or is unavailable, buyer falls back to a guided JSON form in the CLI/frontend.

## Full E2E Data Flow

1. **Buyer defines data need**: Buyer types a natural language query ("I need NestJS backend behavior logs with at least 5,000 rows") into the CLI/dashboard.
2. **LLM translation**: `QueryTranslator` converts the query into a `DataNeedSpec` Protobuf: `{ data_type: "api_behavior_log", min_rows: 5000, required_columns: ["timestamp","endpoint","status_code","latency_ms"], max_null_rate: 0.05, budget_usdc: 10.0 }`.
3. **P2P broadcast**: Buyer agent publishes `DataNeedSpec` to the GossipSub topic `data-need-v1` on the Libp2p network.
4. **Seller discovery**: Seller agents subscribed to `data-need-v1` receive the broadcast and compare it against their local data catalog using semantic embeddings.
5. **Quality pre-scoring**: For matching datasets, seller agent runs the local SLM quality pre-scorer against a 100-row sample. If score < threshold, the dataset is not offered.
6. **ZK proof generation**: Seller agent feeds the full dataset into the circom ZK circuit. The circuit outputs a Groth16 proof attesting to: row count, null rates, schema hash match. This takes 10–60 seconds depending on dataset size.
7. **Offer message**: Seller agent sends a `DataOffer` Protobuf to the buyer: `{ dataset_id, price_usdc, zk_proof, public_inputs (stats only), seller_wallet, data_description }`.
8. **Human approval gate**: Seller-side desktop/CLI notification: "Buyer 0xABC wants to purchase your NestJS logs for 10 USDC. ZK-verified: 8,200 rows, schema ✓, null rate 1.8%. [APPROVE / REJECT]". Human must explicitly approve.
9. **Escrow lock**: Buyer's agent calls `EscrowContract.lockFunds(dataset_id, seller_wallet, price_usdc)` on-chain. Transaction hash recorded.
10. **On-chain ZK verification**: Escrow contract calls the auto-generated Solidity ZK verifier. If proof is valid, escrow state transitions to `VERIFIED`.
11. **Data transfer**: Seller agent generates a one-time AES-256-GCM key, encrypts the full dataset, and streams the encrypted payload to buyer over a direct Libp2p stream. The AES key is encrypted with the buyer's X25519 public key and transmitted separately after the on-chain state is `VERIFIED`.
12. **Settlement**: Buyer agent decrypts, validates locally, and calls `EscrowContract.confirmReceipt()`. Smart contract releases funds to seller wallet.
13. **Fallback**: If buyer does not confirm within N blocks (configurable, default 72 hours), smart contract auto-releases to seller (assuming good faith delivery). If ZK proof fails on-chain, funds auto-refund to buyer.
14. **Logging**: Both agents write a summary record to their local SQLite DBs. The approval event hash is recorded on-chain for audit.

## Privacy & Security

| Concern | Mitigation |
|---------|-----------|
| Raw data exposure to validator | ZK-Proofs verify quality without revealing data content |
| Data interception during transfer | AES-256-GCM encryption; decryption key only released post-settlement |
| Key exchange security | X25519 ECDH (PyNaCl) for key agreement; forward secrecy per transaction |
| Unauthorized data transmission | Hard human-in-the-loop gate; agent cannot proceed without explicit approval event |
| Smart contract exploits | Reentrancy guards, access controls, time-locked fallback, formal audit before mainnet |
| Seller identity exposure | Libp2p PeerID (public key hash) — pseudonymous by design |
| Buyer wallet linkage | Support for burner wallets and stealth addresses (advanced feature) |
| Local data security | SQLite DB encrypted at rest (SQLCipher); data files stored with AES-256 |
| ZK proof forgery | Groth16 is cryptographically sound; on-chain verifier is auto-generated (no custom Solidity logic) |
| Sybil attacks on reputation | Reputation weighted by stake + on-chain transaction history |

## Improvement Suggestions

1. **Layer-2 ZK rollup integration**: Deploy the escrow and ZK verifier on a ZK-rollup chain (zkSync Era or Starknet) so the proof verification itself benefits from the chain's native ZK architecture — reducing gas costs by 10–100x.
2. **Federated ZK proofs for data coalitions**: Allow multiple sellers to jointly prove aggregate statistics over their combined datasets without sharing individual datasets — enabling "data unions" that negotiate as a bloc.
3. **Recursive ZK proofs for data lineage**: Prove that a dataset was derived from a ZK-verified source without revealing the source — enabling a chain of custody for derived datasets.
4. **Reputation NFTs**: On-chain reputation represented as soulbound NFTs (ERC-5192) that accumulate endorsements from successful transactions and cannot be transferred or sold.
5. **Data freshness proofs**: ZK circuits that attest to the recency of the dataset (e.g., "this data was collected within the last 30 days") using signed timestamps from trusted time oracles.
6. **Cross-chain escrow**: Support escrow across chains (e.g., buyer on Ethereum mainnet, seller on Polygon) using cross-chain messaging protocols (LayerZero or Chainlink CCIP).
7. **Differential privacy budget**: Add optional DP noise injection as part of the data preparation pipeline, with the noise parameters attested in the ZK proof — giving buyers a provable privacy guarantee.
8. **Streaming data subscriptions**: Extend the protocol to support ongoing data subscriptions (e.g., "receive new rows every day for 30 days") with periodic micro-payments via state channels.
